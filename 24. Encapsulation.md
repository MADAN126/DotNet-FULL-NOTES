# C# ENCAPSULATION

---

# Why Does Encapsulation Exist?

Before learning syntax like:

```csharp
private decimal balance;

public decimal Balance
{
    get { return balance; }
    set { balance = value; }
}
```

ask a deeper question:

> Why should objects hide their data?

Imagine a bank account class:

```csharp
class BankAccount
{
    public decimal Balance;
}
```

Anyone can do this:

```csharp
account.Balance = -1000000;
```

or

```csharp
account.Balance = 999999999999999999;
```

Problems:

```text
Invalid states become possible.
Business rules are violated.
Bugs become difficult to trace.
Security issues appear.
Any code can break your object.
```

Without encapsulation:

```text
Every object becomes vulnerable.
```

Encapsulation exists because:

> Objects should protect their internal state and control how the outside world interacts with them.

---

# What Is Encapsulation?

Encapsulation means:

> Bundling data and the methods that operate on that data together while restricting direct access to the internal state.

In simpler words:

> Hide what should be hidden.
>
> Expose only what should be used.

---

# Mental Model

Think of an ATM.

You interact with:

```text
Withdraw Money
Deposit Money
Check Balance
```

You DO NOT interact with:

```text
Database queries
PIN encryption algorithms
Cash dispensing mechanisms
Network protocols
```

The ATM hides complexity.

It exposes only safe operations.

Encapsulation works exactly like this.

---

# The Core Philosophy

Encapsulation answers two questions:

## What should outsiders know?

Only what they need.

Example:

```text
Deposit()
Withdraw()
Balance
```

---

## What should outsiders NOT know?

Internal implementation details.

Example:

```text
How balance is stored
Validation logic
Calculation formulas
Database operations
```

# The Problem Without Encapsulation

Example:

```csharp
class Employee
{
    public int Age;
}
```

Usage:

```csharp
Employee emp = new Employee();

emp.Age = -25;
```

Valid?

```text
No.
```

But the compiler allows it.

The object becomes invalid.

---

# Encapsulation Solution

Example:

```csharp
class Employee
{
    private int age;

    public int Age
    {
        get
        {
            return age;
        }

        set
        {
            if (value >= 0)
            {
                age = value;
            }
        }
    }
}
```

---

# Usage

```csharp
Employee emp = new Employee();

emp.Age = 25;

Console.WriteLine(emp.Age);
```

Output:

```text
25
```

---

Trying:

```csharp
emp.Age = -10;
```

Result:

```text
Ignored (or exception if designed that way).
```

The object protects itself.

---

# Internal Working

Execution:

```csharp
emp.Age = 25;
```

Actually becomes:

```text

↓
Validation Executes
↓
Value Approved
↓

```

---

Reading:

```csharp
Console.WriteLine(emp.Age);
```

Becomes:

```text

↓

```

# Encapsulation Through Methods

Encapsulation isn't only properties.

Methods enforce business rules.

Example:

```csharp
class BankAccount
{
    private decimal balance;

    public void Deposit(decimal amount)
    {
        if (amount > 0)
        {
            balance += amount;
        }
    }

    public void Withdraw(decimal amount)
    {
        if (amount > 0 &&
            amount <= balance)
        {
            balance -= amount;
        }
    }
}
```

---

# Why Is This Better?

Users cannot do:

```csharp
balance = -5000;
```

They must use:

```text
Deposit()
Withdraw()
```

Rules remain protected.

# Advantages of Encapsulation

---

# 1. Protects Object State

Prevents invalid values.

---

# 2. Improves Maintainability

Internal implementation can change.

Public interface remains stable.

---

# 3. Reduces Coupling

Other classes depend less on internals.

---

# 4. Supports Validation

Rules are enforced centrally.

---

# 5. Improves Security

Sensitive data stays hidden.

---

# 6. Easier Debugging

All modifications happen through controlled points.

---

# Disadvantages

---

# 1. Slightly More Code

Compared to public fields.

---

# 2. Over-Encapsulation

Too many restrictions can make APIs harder to use.

---

# 3. Poor Design Can Hide Useful Information

Balance is important.

Encryption keys are not.

Judgment matters.

# Interview Perspective

## What is encapsulation?

Answer:

Encapsulation is the process of bundling data and methods together while restricting direct access to internal state.

---

## Why is encapsulation important?

Answer:

It protects object integrity, enforces business rules, improves maintainability, and reduces coupling.

---

## How is encapsulation achieved in C#?

Answer:

Using access modifiers, private fields, public properties, and methods.

---

## Why avoid public fields?

Answer:

Because they expose internal state and allow invalid modifications.

---

## Difference between fields and properties?

Answer:

Fields store data directly, while properties provide controlled access with optional validation.

---

## Is encapsulation the same as information hiding?

Answer:

Not exactly. Encapsulation controls access, while information hiding conceals implementation details. Encapsulation helps achieve information hiding.

---

# Best Practices Followed by Experienced Developers

- Default fields to `private`.
- Expose behavior instead of raw data when possible.
- Use properties rather than public fields.
- Validate values at the point of modification.
- Keep business rules inside the class.
- Use read-only properties for immutable data.
- Avoid exposing mutable collections directly.
- Design public APIs carefully because they become contracts.
