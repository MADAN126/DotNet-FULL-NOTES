# C# EXCEPTIONS — TRY, CATCH, FINALLY, THROW

---

# Why Do Exceptions Exist?

Before learning syntax like:

```csharp
try
{
}
catch
{
}
```

ask a deeper question:

> What happens when something goes wrong while a program is running?

Imagine this code:

```csharp
int number = int.Parse("abc");

Console.WriteLine(number);
```

The user entered:

```text
abc
```

instead of:

```text
123
```

What should happen?

Option 1:

```text
Program crashes immediately.
```

Option 2:

```text
Program detects the problem,
handles it gracefully,
and continues running.
```

Which would users prefer?

Obviously:

```text
Option 2.
```

Exceptions exist because:

> Programs operate in an unpredictable world where things fail, and software needs a structured way to detect, communicate, and recover from those failures.

---

# What Is an Exception?

An exception is:

> An object that represents an error or unexpected situation occurring during program execution.

---

# Simple Definition

An exception means:

```text
"Something went wrong,
and normal execution cannot continue as expected."
```

---

# Mental Model

```text

↓

↓

```

Unexpected event:

```text
Flat Tire
```

Now normal driving stops.

You must handle the problem.

Software works the same way.

# Catching Specific Exceptions

One of the most important practices.

---

# Why?

Bad:

```csharp
catch
{
}
```

This catches everything.

Sometimes too broad.

---

# Better

```csharp

{
}
```

---

# Example

```csharp

{
    int num = 10 / 0;
}
catch (DivideByZeroException)
{
    Console.WriteLine("Cannot divide by zero.");
}
```

Output:

```text
Cannot divide by zero.
```

Example:

```csharp

}
```

```text

Specific catch becomes unreachable.
```

Compiler error.

```text

↓
Most General
```

Example:

```csharp

{
}

{
}

{
}
```

# What Is throw?

The `throw` keyword:

> Explicitly creates or rethrows exceptions.

---

# Why?

Sometimes your code detects invalid situations.

---

# Example

```csharp
if(age < 0)
{
    throw new ArgumentException(
        "Age cannot be negative.");
}
```

---

# Mental Model

Think:

```text
"I cannot handle this.
Someone else must know."
```

---

# Example

```csharp
try
{
    throw new Exception("Manual error");
}
catch(Exception ex)
{
    Console.WriteLine(ex.Message);
}
```

Output:

```text
Manual error
```

---

# Rethrowing Exceptions

Wrong:

```csharp
catch(Exception ex)
{
    throw ex;
}
```

---

# Problem

Stack trace resets.

---

# Better

```csharp
catch(Exception)
{
    throw;
}
```

---

# Why?

Preserves original debugging information.

# Exceptions Are Expensive

Throwing exceptions:

```text
Allocates objects.
Captures stack traces.
Unwinds call stack.
```

Use for:

```text
Exceptional situations.
```

Not:

```text
Normal program flow.
```

---

# TryParse vs Exceptions

Bad:

```csharp
try
{
    int.Parse(input);
}
catch
{
}
```

---

# Better

```csharp
if(int.TryParse(input, out int number))
{
    Console.WriteLine(number);
}
else
{
    Console.WriteLine("Invalid input");
}
```

---

# Why?

Because invalid input is expected.

Not exceptional.

# Interview Perspective

## What is an exception?

Answer:

An exception is an object representing an error that disrupts normal program execution.

---

## Why use try-catch?

Answer:

To handle runtime errors gracefully without crashing the application.

---

## What is finally?

Answer:

A block that always executes, typically used for cleanup.

---

## Difference between throw and throw ex?

Answer:

`throw` preserves the original stack trace, while `throw ex` resets it.

---

## Why shouldn't exceptions be used for normal flow?

Answer:

Because throwing exceptions is expensive and expected scenarios should be handled using normal control structures.

---

## Why is TryParse preferred over Parse in user input?

Answer:

Because invalid input is expected and TryParse avoids exception overhead.

---

# Best Practices Followed by Experienced Developers

- Catch only exceptions you can meaningfully handle.
- Prefer specific exceptions over generic ones.
- Never swallow exceptions silently.
- Use `finally` or `using` for cleanup.
- Preserve stack traces using `throw`.
- Use `TryParse` for expected validation failures.
- Log exceptions with sufficient context.
- Treat exceptions as exceptional events, not control flow mechanisms.

---

# Final Mental Model

Think of exceptions as emergency protocols.

```text
Normal Execution
↓
Unexpected Failure
↓
Exception Object Created
↓
Call Stack Unwinds
↓
Matching Catch Found
↓
Recovery or Logging Happens
↓
Finally Cleans Resources
↓
Program Continues or Terminates Safely
```

A beginner sees:

```csharp
try
{
}
catch
{
}
```

as syntax.

An experienced engineer sees:

> A structured error-handling mechanism that separates normal behavior from failure scenarios, preserves diagnostic information, enables graceful recovery, and allows complex systems to remain reliable in the face of inevitable runtime problems.
