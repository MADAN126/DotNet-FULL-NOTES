# C# TYPE CASTING

---

# Why Does Type Casting Exist?

Before learning syntax, understand the real problem.

Programs constantly move data from one form to another.

Examples:

```text
User enters age as text        → "25"
Database stores amount as int  → 100
Calculation produces decimal   → 99.99
API returns object             → Actual type hidden
```

These values often need to become different types.

Without type casting, data would be trapped inside its original type.

Example:

```csharp
int age = 25;

double result = age;
```

Why should this work?

Or:

```csharp
double price = 99.99;

int wholePrice = ???;
```

How should C# handle it?

Type casting exists because:

> Real-world data constantly changes representation.

---

# What Is Type Casting?

Type casting is the process of converting a value from one data type to another.

Example:

```csharp
int age = 25;

double ageDouble = age;
```

Here:

```text
int
↓
double
```

The value changes representation.

---

# Mental Model

Think of type casting as pouring liquids into different containers.

Example:

```text
Small Glass
↓
Large Jug
```

Easy.

No information lost.

But:

```text
Large Jug
↓
Small Glass
```

Some liquid may spill.

Information may be lost.

This is the entire philosophy behind type casting.

---

# Why Does Type Casting Exist?

Because different systems speak different "data languages."

Examples:

```text
Database       → int
UI             → string
Math Engine    → double
Finance Module → decimal
API            → object
```

Casting acts as a translator.

---

# Categories of Type Casting

C# supports two major forms:

```text
1. Implicit Casting
2. Explicit Casting
```

---

# Overview

```text
Type Casting
│
├── Implicit (Automatic)
│
└── Explicit (Manual)
```

# IMPLICIT CASTING

---

# What Is Implicit Casting?

Implicit casting happens automatically.

The compiler performs it for you.

Example:

```csharp
int age = 25;

double ageDouble = age;
```

---

# Mental Model

Think:

```text
Small Container
↓
Large Container
```

Safe transfer.

---

# Why Does It Exist?

Because some conversions cannot lose information.

Example:

```text
int → long
int → double
char → int
```

The destination can hold everything the source contains.

---

# Example

```csharp
int number = 100;

double value = number;

Console.WriteLine(value);
```

Output:

```text
100
```

---

# Internal Working

Execution:

```text
Compiler checks compatibility
↓
No data loss possible
↓
Automatic conversion inserted
↓
IL generated
↓
CLR executes
```

---

# Common Implicit Conversions

```text
char
 ↓
int
 ↓
long
 ↓
float
 ↓
double
 ↓
decimal
```

---

# Example

```csharp
char letter = 'A';

int ascii = letter;

Console.WriteLine(ascii);
```

Output:

```text
65
```

---

# Why?

Characters internally store Unicode values.

'A' corresponds to:

```text
65
```

---

# Advantages

- Simple
- Safe
- Readable
- Prevents unnecessary syntax

---

# Disadvantages

- Can hide performance costs in rare cases
- Developers may forget conversions occur

---

# EXPLICIT CASTING

---

# What Is Explicit Casting?

Explicit casting requires the developer to request conversion manually.

Example:

```csharp
double price = 99.99;

int wholePrice = (int)price;
```

---

# Mental Model

Think:

```text
Large Container
↓
Small Container
```

Risky.

Something might not fit.

---

# Why Does It Exist?

Because the compiler cannot safely assume information loss is acceptable.

Only you can decide.

---

# Example

```csharp
double temperature = 36.9;

int rounded = (int)temperature;

Console.WriteLine(rounded);
```

Output:

```text
36
```

---

# Hidden Behavior

Beginners often think:

> It rounds.

Reality:

It truncates.

---

# Truncation

Example:

```csharp
(double)36.9
↓
(int)36.9
↓
36
```

Fraction removed.

Not rounded.

---

# Internal Working

Execution:

```text
Compiler detects explicit cast
↓
Allows potentially unsafe conversion
↓
CLR performs conversion
↓
Excess information discarded
```

---

# Advantages

- Gives developer control
- Allows otherwise impossible conversions

---

# Disadvantages

- Data loss possible
- Requires caution

---

# Implicit vs Explicit Casting

| Feature | Implicit | Explicit |
|----------|-----------|-----------|
| Automatic | Yes | No |
| Syntax Needed | No | Yes |
| Data Loss Risk | No | Yes |
| Compiler Approval | Automatic | Requires Cast |
| Safety | High | Lower |
| Example | int → double | double → int |

---

# Understanding Data Loss

Consider:

```csharp
double value = 99.99;

int result = (int)value;
```

Memory:

```text
99.99
↓
Remove fractional part
↓
99
```

Lost:

```text
0.99
```

Gone forever.

---

# Real-World Analogy

Implicit:

```text
Moving books from a backpack to a truck.
```

Everything fits.

---

Explicit:

```text
Moving furniture into a backpack.
```

Some items must be discarded.

# TryParse

One of the most important methods in real projects.

---

# Why Does TryParse Exist?

Users make mistakes.

Example:

User enters:

```text
abc
```

Instead of:

```text
25
```

What should happen?

Crash?

No.

---

# Example

```csharp
string input = "25";

bool success = int.TryParse(input, out int age);

Console.WriteLine(success);
Console.WriteLine(age);
```

Output:

```text
True
25
```

---

# Invalid Input

```csharp
string input = "abc";

bool success = int.TryParse(input, out int age);
```

Output:

```text
False
0
```

---

# Mental Model

Parse:

```text
Convert this.
If impossible, crash.
```

TryParse:

```text
Try.
If impossible, tell me safely.
```

---

# Why Senior Developers Prefer TryParse

Because applications interact with humans.

Humans provide bad input.

Robust systems anticipate failure.

# Hidden Behaviors and Edge Cases

---

# Explicit Casting Does NOT Round

Example:

```csharp
(int)9.99
```

Result:

```text
9
```

Not:

```text
10
```

---

# Overflow Can Occur

Example:

```csharp
long big = 3000000000;

int small = (int)big;
```

Unexpected results possible.

---

# Convert Performs Rounding

Example:

```csharp
Convert.ToInt32(9.7);
```

Output:

```text
10
```

---

# Compare

```csharp
(int)9.7
```

Output:

```text
9
```

---

# Important Difference

| Conversion | Result |
|-------------|---------|
| `(int)9.7` | 9 |
| `Convert.ToInt32(9.7)` | 10 |

---

# Parse Throws Exceptions

Example:

```csharp
int.Parse("ABC");
```

Runtime Exception.

---

# TryParse Does Not

Example:

```csharp
int.TryParse("ABC", out int value);
```

Output:

```text
False
```

Program continues.

# Interview Perspective

## What is type casting?

Answer:

Type casting is converting a value from one data type to another.

---

## Difference between implicit and explicit casting?

Answer:

Implicit casting occurs automatically when safe. Explicit casting requires developer intervention because data loss may occur.

---

## Does explicit casting round values?

Answer:

No. It truncates the fractional part.

---

## Difference between Parse and TryParse?

Answer:

Parse throws exceptions on invalid input, whereas TryParse safely returns a boolean indicating success or failure.

---

## Why use TryParse?

Answer:

It prevents applications from crashing due to invalid user input.

---

## Difference between Convert.ToInt32() and explicit casting?

Answer:

Explicit casting truncates decimals, while Convert.ToInt32() rounds to the nearest integer.

---

# Best Practices Followed by Experienced Developers

- Prefer implicit conversions when available.
- Use explicit casting carefully.
- Never assume casting rounds values.
- Use decimal for financial data.
- Prefer TryParse for user input.
- Validate external data before conversion.
- Be aware of overflow risks.
- Understand where information loss can occur.

---

Think of type casting as translation between data worlds.

```text
Data Exists
↓
New Representation Needed
↓
Can It Be Converted Safely?
        ↓
      Yes
        ↓
Implicit Casting
        ↓
Automatic Conversion

        OR

Potential Information Loss
        ↓
Explicit Casting
        ↓
Developer Approval Required

        ↓

        ↓

```

A beginner sees:

```csharp

```

as syntax.

An experienced engineer sees:

> A deliberate conversion decision involving type safety, potential data loss, runtime behavior, and business requirements, chosen carefully to preserve correctness while enabling systems to communicate across different representations of data.
