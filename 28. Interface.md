# C# INTERFACES & MULTIPLE INTERFACES

---

# Why Do Interfaces Exist?

Before learning syntax like:

```csharp
interface ILogger
{
    void Log(string message);
}
```

ask a deeper question:

> Why should classes agree on behavior without sharing implementation?

Imagine an online payment system.

You have:

```text
UPI Payment
Credit Card Payment
PayPal Payment
Net Banking Payment
```

All of them can:

```text
Process payments.
```

But internally they work very differently.

Without interfaces:

```csharp
class PaymentProcessor
{
    public void Process(UPIPayment payment)
    {
        payment.Pay();
    }
}
```

Now management says:

> "Add Credit Card support."

You modify code.

Then:

> "Add PayPal."

Modify again.

Then:

> "Add Apple Pay."

Modify again.

Problems:

```text
Tightly coupled code.
Endless modifications.
Poor extensibility.
Difficult testing.
Violates Open/Closed Principle.
```

Interfaces exist because:

> Software should depend on capabilities, not concrete implementations.

---

# What Is an Interface?

An interface is:

> A contract that specifies WHAT a class must do without specifying HOW it does it.

---

# Simple Definition

An interface says:

```text
"If you claim to implement me,
you MUST provide these behaviors."
```

---

# Mental Model

Think of a power socket.

The socket defines:

```text
Voltage standard
Pin arrangement
Connection rules
```

It does NOT define:

```text
TV
Laptop
Phone Charger
Microwave
```

Different devices plug into the same socket.

Same contract.

Different implementations.

---

# Core Philosophy

Instead of asking:

```text
"Who are you?"
```

we ask:

```text
"What can you do?"
```

---

# Example

Instead of:

```text
Is this UPIPayment?
Is this CreditCardPayment?
```

We ask:

```text
Can you Pay()?
```

If yes:

Use it.

# Implementing an Interface

Interface:

```csharp
interface IAnimal
{
    void MakeSound();
}
```

Implementation:

```csharp
class Dog : IAnimal
{
    public void MakeSound()
    {
        Console.WriteLine("Bark");
    }
}
```

---

# Usage

```csharp
Dog dog = new Dog();

dog.MakeSound();
```

Output:

```text
Bark
```

---

# What Happens Internally?

Execution:

```text
Dog says:
"I implement IAnimal."

↓

Compiler checks:

Does Dog implement MakeSound()?

↓

Yes

↓

Compilation succeeds.
```

If not:

Compilation fails.

Interfaces:

```text

Contain contracts.
Can declare methods.
Can declare properties.

Support multiple inheritance.
```

# Interfaces and Polymorphism

One of their biggest strengths.

Interface:

```csharp
interface IPayment
{
    void Pay(decimal amount);
}
```

Implementations:

```csharp

{
    public void Pay(decimal amount)
    {
        Console.WriteLine("UPI");
    }
}

class CreditCardPayment : IPayment
{
    public void Pay(decimal amount)
    {
        Console.WriteLine("Credit Card");
    }
}
```

---

```csharp

payment = new UPIPayment();

payment.Pay(500);

payment = new CreditCardPayment();

payment.Pay(500);
```

Output:

```text
UPI
Credit Card
```

# Abstract Classes vs Interfaces

Extremely common interview topic.

---

# Comparison Table

| Feature | Abstract Class | Interface |
|-----------|---------------|------------|
| Fields | Yes | No |
| Constructors | Yes | No |
| Shared State | Yes | No |
| Method Implementation | Yes | Limited* |
| Multiple Inheritance | No | Yes |
| Purpose | Base Type | Capability Contract |
| Access Modifiers | Yes | Public by contract |
| Instantiation | No | No |

*Modern C# supports default interface implementations.

---

# When Should You Use Interfaces?

Use interfaces when:

```text
You want loose coupling.
You need multiple capabilities.
You expect multiple implementations.
You want easier testing.
You design contracts.
```

---

# When Should You Avoid Interfaces?

Avoid interfaces when:

```text
Only one implementation exists forever.
You need shared fields.
You need constructors.
You need substantial shared implementation.
```

# Interview Perspective

## What is an interface?

Answer:

An interface is a contract that defines what a class must do without specifying how it does it.

---

## Why use interfaces?

Answer:

They promote loose coupling, extensibility, testability, and polymorphism.

---

## Can interfaces be instantiated?

Answer:

No.

---

## Can a class implement multiple interfaces?

Answer:

Yes.

This is one of the major advantages of interfaces.

---

## Difference between abstract classes and interfaces?

Answer:

Abstract classes provide shared state and implementation, while interfaces define capabilities and support multiple implementations.

---

## Why doesn't C# support multiple class inheritance?

Answer:

To avoid ambiguity problems such as the Diamond Problem.

Interfaces avoid this because they traditionally don't carry implementation state.

---

# Best Practices Followed by Experienced Developers

- Program against interfaces, not concrete classes.
- Create interfaces when multiple implementations or testing needs exist.
- Keep interfaces focused and cohesive.
- Prefer small capability-based interfaces.
- Avoid interface explosion.
- Use interfaces heavily with Dependency Injection.
- Choose abstract classes when shared implementation is important.
- Design interfaces as stable contracts.
