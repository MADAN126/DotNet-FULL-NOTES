# C# COMMENTS

---

# Why Do Comments Exist?

Imagine joining a company and opening a codebase with 50,000 lines of code.

You see this:

```csharp
int x = amount * 18 / 100;
```

Questions immediately arise:

- Why 18?
- Is it a tax rate?
- Is it a discount?
- Is it a business rule?
- Can I change it?
- What happens if I remove it?

The code tells you **what** it does.

It rarely tells you **why** it exists.

Comments exist to bridge that gap.

> Code explains HOW.
>
> Comments should explain WHY.

---

# What Are Comments?

Comments are notes written inside source code that are ignored by the compiler.

They exist purely for humans.

Example:

```csharp
// Calculate GST at 18%
int gst = amount * 18 / 100;
```

The compiler executes:

```csharp
int gst = amount * 18 / 100;
```

It completely ignores:

```csharp
// Calculate GST at 18%
```

---

# Mental Model

Think of comments as sticky notes attached to code.

The machine never reads them.

Future developers do.

Including:

- Your teammates
- Future maintainers
- Interviewers reviewing take-home assignments
- Open-source contributors
- Future you

The most important reader is often:

> You, six months later.

---

# What Problem Do Comments Solve?

Without comments:

```csharp
int x = amount * 18 / 100;
```

Meaning:

```text
???
```

With comments:

```csharp
// GST mandated by government regulations
int gst = amount * 18 / 100;
```

Meaning:

```text
Business Rule
```

Comments provide context.

---

# What Happens If Comments Did Not Exist?

Developers would rely solely on code.

This sounds ideal until reality appears.

Some decisions cannot be expressed through syntax.

For example:

- Legal requirements
- Temporary workarounds
- Performance trade-offs
- External API quirks
- Historical reasons
- Business decisions

Without comments:

Teams waste time rediscovering old decisions.

---

# Internal Working

Consider:

```csharp
// Print welcome message
Console.WriteLine("Welcome");
```

Compilation Flow:

```text
Source Code
↓
Compiler Reads File
↓
Comment Detected
↓
Comment Ignored
↓
IL Generated Only For Executable Code
↓
CLR Executes Program
```

The generated program contains no comment logic.

Comments do not affect execution.

---

# Types of Comments in C#

C# primarily supports three kinds of comments.

```text
1. Single-Line Comments
2. Multi-Line Comments
3. XML Documentation Comments
```

Each exists for a different reason.

---

# 1. Single-Line Comments

Syntax:

```csharp
// comment
```

---

# What Is It?

A comment that starts with:

```csharp
//
```

Everything after it on that line becomes a comment.

---

# Example

```csharp
// Print greeting
Console.WriteLine("Hello");
```

---

# Internal Working

Compiler sees:

```csharp
//
```

Then ignores everything until:

```text
End of Line
```

---

# Mental Model

Think of it as highlighting one sentence.

Example:

```text
NOTE: This line explains the next instruction.
```

---

# When Should You Use It?

Use for:

- Explaining intent
- Clarifying business rules
- Adding brief notes
- Describing tricky logic

Example:

```csharp
// Retry three times because partner API occasionally times out
```

---

# When Should You Avoid It?

Avoid obvious comments.

Bad:

```csharp
// Increment i
i++;
```

The code already explains itself.

---

# Advantages

- Short
- Easy to read
- Commonly used
- Minimal clutter

---

# Disadvantages

- Limited to one line
- Excessive usage can create noise

---

# Common Beginner Mistakes

Writing comments that merely translate English into code.

Example:

```csharp
// Set age to 20
int age = 20;
```

Unnecessary.

---

# Better Version

```csharp
// Minimum age required by insurance regulations
int age = 20;
```

Now the reason is documented.

---

# 2. Multi-Line Comments

Syntax:

```csharp
/*
comment
*/
```

---

# What Is It?

A comment spanning multiple lines.

---

# Example

```csharp
/*
This calculation follows
the pricing model approved
by the finance department.
*/
decimal total = CalculatePrice();
```

---

# Internal Working

Compiler ignores everything between:

```csharp
/*
```

and

```csharp
*/
```

---

# Mental Model

Think of it as a paragraph in a book.

Single-line comments:

```text
Sticky note.
```

Multi-line comments:

```text
Detailed explanation.
```

---

# Why Does It Exist?

Some explanations cannot fit into one line.

Examples:

- Complex algorithms
- Legal requirements
- Integration notes
- Temporary workarounds

---

# Advantages

- Detailed explanations
- Better for large notes
- Improves understanding of complex logic

---

# Disadvantages

- Can become outdated
- Large blocks reduce readability
- Temptation to over-explain

---

# Common Beginner Mistakes

Using multi-line comments to disable code.

Example:

```csharp
/*
Console.WriteLine("Test");
Console.WriteLine("Debug");
*/
```

---

# Why Is This Dangerous?

Because:

- Old code accumulates
- Nobody knows whether it is safe to delete
- Version control already preserves history

---

# Best Practice

Delete unused code.

Git remembers it.

---

# 3. XML Documentation Comments

Syntax:

```csharp
/// comment
```

This is the most misunderstood comment type.

---

# What Is It?

Structured comments that generate documentation.

Example:

```csharp
/// <summary>
/// Calculates GST amount.
/// </summary>
decimal CalculateGST(decimal amount)
{
    return amount * 0.18m;
}
```

---

# Why Does It Exist?

Large teams build reusable libraries.

Developers need documentation directly inside IDEs.

Example:

When another developer types:

```csharp
CalculateGST(
```

They immediately see:

```text
Calculates GST amount.
```

Without opening documentation websites.

---

# Internal Working

Flow:

```text
XML Comment
↓
Compiler Reads XML
↓
Documentation File Generated
↓
IDE Displays Help
↓
Developers Understand APIs Faster
```

---

# Mental Model

Think of XML comments as:

```text
Code + User Manual
```

living together.

---

# Example

```csharp
/// <summary>
/// Adds two numbers.
/// </summary>
/// <param name="a">First number.</param>
/// <param name="b">Second number.</param>
/// <returns>The sum.</returns>
int Add(int a, int b)
{
    return a + b;
}
```

---

# Common XML Tags

| Tag | Purpose |
|----------|----------|
| `<summary>` | Explains what member does |
| `<param>` | Describes parameters |
| `<returns>` | Describes return value |
| `<exception>` | Documents exceptions |
| `<remarks>` | Additional details |
| `<example>` | Usage examples |
| `<see>` | Cross references |

---

# Why Not Comment Everything?

Many beginners think:

> More comments = better code.

Reality:

Bad comments hurt more than no comments.

---

# Example

Bad:

```csharp
// Create customer
Customer customer = new Customer();
```

The code already explains this.

---

# Better

```csharp
// Customer IDs come from the external CRM system
Customer customer = new Customer();
```

Explains hidden knowledge.

---

# Code vs Comments

Good code:

```csharp
decimal gstAmount = amount * 0.18m;
```

Bad code requiring comments:

```csharp
decimal x = amount * 0.18m; // GST
```

---

# Senior Developer Philosophy

First:

```text
Write self-explanatory code.
```

Then:

```text
Comment only what code cannot express.
```

# What Should Comments Explain?

Good comments explain:

---

## Why

Example:

```csharp
// Required because payment gateway rejects lowercase currency codes
currency = currency.ToUpper();
```

---

## Business Rules

Example:

```csharp
// Interest rates fixed by RBI guidelines
```

---

## Workarounds

Example:

```csharp
// Temporary workaround until vendor fixes API issue
```

---

## Performance Decisions

Example:

```csharp
// Cached to avoid repeated database queries
```

---

## Security Considerations

Example:

```csharp
// Token intentionally expires after 5 minutes
```

---

# What Should Comments NOT Explain?

Avoid comments that repeat code.

Bad:

```csharp
// Loop through users
foreach(var user in users)
{
}
```

---

Bad:

```csharp
// Return result
return total;
```

---

Bad:

```csharp
// Call Save method
Save();
```

# Common Beginner Mistakes

---

## Commenting Every Line

Bad:

```csharp
// Declare variable
int age;

// Assign value
age = 20;
```

Creates noise.

---

## Leaving Dead Code

Bad:

```csharp
// Old implementation
/*
CalculateLegacyTax();
*/
```

Git already tracks history.

---

## Writing Outdated Comments

Example:

Comment:

```csharp
// Applies 10% discount
```

Code:

```csharp
return amount * 0.20m;
```

Now comments lie.

---

## Using Comments Instead of Refactoring

Bad:

```csharp
// Complex pricing logic begins
```

followed by 300 lines.

Better:

Extract methods.

---

# Comments vs Naming

Consider:

Bad naming:

```csharp
int x;
```

Needs comments.

---

Better naming:

```csharp
int maximumLoginAttempts;
```

Comment unnecessary.

|----------|-------------|------------|-------------------|
| Syntax | `//` | `/* */` | `///` |
| Purpose | Short notes | Detailed explanations | API documentation |
| Compiler Execution | Ignored | Ignored | Generates docs |
| Scope | One line | Multiple lines | Members |
| IDE Support | Minimal | Minimal | Excellent |
| Enterprise Usage | Very High | Moderate | Very High |
| Best For | Intent | Detailed context | Public APIs |

# Hidden Behaviors and Edge Cases

---

## Comments Are Removed During Compilation

Compiled programs do not execute comments.

---

## XML Comments Can Generate Warnings

Missing documentation may trigger warnings in some projects.

Example:

```text
CS1591
```

---

## Nested Multi-Line Comments Are Not Allowed

Incorrect:

```csharp
/*
Outer

/*
Inner
*/

*/
```

Compiler error.

---

# Why?

Compiler cannot determine where comments end.

---

## XML Comments Improve IntelliSense

Developers hovering over methods see documentation instantly.

This significantly improves developer experience.

---

# Interview Perspective

## What are comments in C#?

Answer:

Comments are non-executable notes used to explain code and are ignored by the compiler.

---

## What are the types of comments?

Answer:

- Single-line comments (`//`)
- Multi-line comments (`/* */`)
- XML documentation comments (`///`)

---

## What is the purpose of XML comments?

Answer:

They provide structured documentation that IDEs and generated documentation tools can use.

---

## Are comments compiled?

Answer:

Regular comments are ignored during compilation. XML comments may generate documentation metadata but do not become executable instructions.

---

## Why are excessive comments considered bad?

Answer:

They create noise, become outdated, and often compensate for poorly written code instead of improving clarity.

---

# Best Practices Followed by Experienced Developers

- Prefer expressive code over explanatory comments.
- Comment the "why," not the "what."
- Keep comments synchronized with code changes.
- Use XML comments for public APIs and libraries.
- Delete obsolete commented-out code.
- Explain business rules and unusual decisions.
- Refactor confusing code before adding comments.
- Treat comments as part of the codebase that requires maintenance.

---

# Final Mental Model

Think of comments as communication layers.

```text
Code
↓
Explains HOW something works

Naming
↓
Explains WHAT something represents

Comments
↓
Explain WHY decisions were made

XML Documentation
↓
Teaches OTHER developers how to use your APIs

Version Control
↓
Preserves historical changes
```

A junior developer often uses comments to narrate code.

A senior developer uses comments to preserve knowledge that cannot be discovered by reading the code itself.

> Good code tells you what it does.
>
> Great comments tell you why it had to be done that way.
