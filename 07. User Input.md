# C# USER INPUT

---

# Why Does User Input Exist?

Before learning syntax, understand the problem.

Imagine software that cannot receive information from users.

Examples:

A calculator:

```text
Can only add 5 + 10 forever.
```

A login page:

```text
Cannot accept usernames.
```

A banking application:

```text
Cannot receive transfer amounts.
```

An online shopping site:

```text
Cannot know what users want to buy.
```

Such software would be useless.

Programs become powerful because they can react to information provided during execution.

User input exists because:

> Software must adapt to changing information instead of operating on hardcoded values.

---

# What Is User Input?

User input is data provided by the user while the program is running.

Example:

```text
Enter your name:
John
```

The program receives:

```text
John
```

and uses it.

---


With input:

```text

```

Real software depends heavily on this interaction.



The computer cannot know.

Therefore:

> Console.ReadLine() always returns a string.

The developer decides how to interpret it.

# Console.Write vs Console.WriteLine

Many beginners confuse these.

---

# Console.WriteLine()

Moves to the next line.

Example:

```csharp
Console.WriteLine("Hello");
Console.WriteLine("World");
```

Output:

```text
Hello
World
```

---

# Console.Write()

Stays on the same line.

Example:

```csharp
Console.Write("Hello ");
Console.Write("World");
```

Output:

```text
Hello World
```

---

# Comparison

| Feature | Write | WriteLine |
|----------|--------|------------|
| New Line Added | No | Yes |
| Prompt Friendly | Yes | Sometimes |
| Most Common | Moderate | Very High |

```text

```

```text
abc
```



Program crashes.

---

# Why Does This Happen?

Execution:

```text
ReadLine
↓
Returns "abc"
↓
Convert.ToInt32()
Attempts Conversion
↓
Conversion Impossible
↓
Exception Thrown
```

---

# The Problem with Convert and Parse

They assume:

> The user entered valid input.

Reality:

Users make mistakes.

---

# TryParse: The Professional Approach

Example:

```csharp
Console.Write("Enter age: ");

string input = Console.ReadLine();

bool success = int.TryParse(input, out int age);

if (success)
{
    Console.WriteLine("Age: " + age);
}
else
{
    Console.WriteLine("Invalid input.");
}
```

---

# Execution Flow

Valid Input:

```text
25
↓

success = true
↓

age = 25
```

---

Invalid Input:

```text
abc
↓

success = false
↓

Program continues safely.
```

---

# Why Senior Developers Prefer TryParse

Because:

```text
Users are unpredictable.
```

Robust systems expect failure.

# Interview Perspective

## What does Console.ReadLine() do?

Answer:

It reads an entire line of user input from the console and returns it as a string.

---

## Why does ReadLine return a string?

Answer:

Because the console cannot determine the intended data type of user input, so it returns raw text for the developer to interpret.

---

## Difference between Write and WriteLine?

Answer:

Write outputs text without moving to the next line, while WriteLine outputs text and then moves to a new line.

---

## Why is TryParse preferred?

Answer:

Because it safely handles invalid input without throwing exceptions, making applications more robust.

---

## What happens if invalid input is passed to Convert.ToInt32()?

Answer:

A runtime exception is thrown if the string cannot be converted to an integer.

---

## What is the Input-Process-Output model?

Answer:

A fundamental programming model where programs receive input, process it, and produce output.

---

# Best Practices Followed by Experienced Developers

- Always display clear prompts.
- Assume users will provide invalid input.
- Prefer TryParse for user-entered data.
- Validate ranges and business rules after conversion.
- Trim unnecessary whitespace.
- Separate input, processing, and output logic.
- Never trust external input blindly.
- Design programs to fail gracefully.

Think of user input as a conversation between humans and software.

```text
Program Asks
↓
User Responds
↓
ReadLine Captures Raw Text
↓
Conversion Interprets Meaning
↓
Validation Ensures Correctness
↓
Business Logic Processes Data
↓
Output Communicates Results
```

A beginner sees:

as syntax.

An experienced engineer sees:

> The boundary where unpredictable human input enters the system, requiring careful conversion, validation, error handling, and defensive programming to transform raw text into trustworthy data that the application can safely use.
